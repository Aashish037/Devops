# Domain Model

## Purpose

Defines the core business concepts of the Workspace Management SaaS platform.

This document establishes the domain vocabulary used throughout the system and serves as the foundation for relationship modeling, ownership modeling, database design, and API design.

This document is implementation-agnostic.

---

# Tenant Boundary

## Decision

An Organization represents the tenant boundary.

All business entities operate within an organizational context.

## Requirements

* Tenant data must remain isolated.
* Cross-tenant access is prohibited.
* Authorization decisions must be tenant-aware.

---

# Core Domain Entities

## Organization

Represents a tenant within the platform.

---

## User

Represents an authenticated individual interacting with the platform.

---

## Membership

Represents the association between a user and an organization.

Memberships establish organizational participation and role assignment.

---

## Role

Represents a predefined set of responsibilities within an organization.

Roles are used to group permissions.

---

## Permission

Represents an authorized action within the platform.

Permissions are assigned through roles and evaluated during authorization.

---

## Team

Represents a logical grouping of users within an organization.

Teams support organizational structure and project ownership.

---

## Project

Represents a business initiative managed within an organization.

Projects act as containers for related work.

---

## Task

Represents an actionable unit of work within a project.

Tasks progress through defined workflow states and are assigned to a single responsible user.

---

## Comment

Represents collaborative communication associated with business entities.

Comments support discussion, review feedback, and project collaboration.

---

## Attachment

Represents a file associated with a business entity.

Attachments support document sharing and workflow execution.

---

## Invitation

Represents an onboarding request issued by an organization.

Invitations are used to establish organizational membership.

---

## Notification

Represents a user-facing event generated by system activity.

Notifications communicate workflow and operational events.

Notification history is retained for user visibility.

---

## Activity Log

Represents an immutable audit record of significant system actions.

Activity logs provide traceability, operational visibility, and security auditing.

---

# Domain Constraints

The following decisions apply to Version 1:

* A user belongs to a single organization.
* A user has a single role within an organization.
* A task has a single assignee.
* A project has a single owner.
* Notifications are persisted.
* Activity logs are persisted.
* Soft deletion is preferred over permanent deletion.

---

# Deferred Entities

The following domain concepts are excluded from the initial release.

## Approval Workflow

Dedicated workflow engine for configurable approvals.

## Department

Hierarchical organizational structure.

## Workflow Automation

Rule-driven workflow execution and orchestration.

## SLA Policy

Escalation, deadline, and service-level management.
